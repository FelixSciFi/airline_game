# 项目开发原则 (PROJECT RULES)

本文档约定本项目的代码结构与开发规范。后续开发必须遵守本规则，避免结构混乱。

---

## 一、项目开发原则

1. **每个功能尽量拆分为独立模块**  
   一个功能对应一个（或一组）清晰边界的文件/目录，避免“万能脚本”。

2. **不要把大量逻辑堆在一个脚本里**  
   单脚本职责单一，便于维护和测试。

3. **UI 逻辑与游戏逻辑尽量分离**  
   界面只负责展示与用户输入；业务/经济/飞行等逻辑放在独立模块（如 `systems/`），由 UI 调用或通过信号通信。

4. **地图逻辑不能包含游戏系统逻辑**  
   `world_map` 只负责地图与城市的展示，以及打开/关闭菜单、切换面板；不包含飞机、航班、乘客、经济等系统代码。

5. **任何新增功能前先检查是否应该新建模块**  
   先判断功能归属（地图 / 菜单 / 财务 / 系统），再决定在现有模块扩展还是新建目录与脚本。

---

## 二、代码结构原则

### world_map 的职责（仅限以下内容）

- 地图显示
- 城市显示
- 打开主菜单、关闭主菜单
- 从主菜单进入账单、从账单返回主菜单（仅做面板的显示/隐藏与信号连接）

**禁止在 world_map.gd 中：**  
写飞机、航班、乘客、经济逻辑，或直接创建账单/其他子界面的 UI 节点（应通过实例化独立场景完成）。

### 目录与模块归属

| 模块         | 路径               | 说明                     |
|--------------|--------------------|--------------------------|
| 主菜单       | `res://ui/main_menu/` | 主菜单面板场景与脚本     |
| 财务相关 UI  | `res://ui/finance/`   | 账单等财务界面           |
| 游戏系统     | `res://systems/`      | 未来：飞机、航班、经济等 |
| 静态数据     | `res://data/`         | 如 cities.json 等         |
| 通用 UI      | `res://ui/`           | 如 world_map             |

新增功能时，必须放入对应目录，不得随意放在根目录或无关脚本中。

---

## 三、文件拆分原则

当出现以下情况时，**必须**拆分为独立文件或模块：

- **单个脚本超过 200 行**  
  考虑按职责拆成多个脚本或子场景。

- **一个脚本承担多个功能**  
  例如既管地图又管经济，应拆成“地图模块”+“经济模块”。

- **UI 和逻辑混在一起**  
  界面层（场景/控件）与业务逻辑（计算、状态）应分离，逻辑放入独立脚本或 `systems/`。

- **一个界面开始管理多个子界面**  
  每个子界面应有自己的场景与脚本，父级只做打开/关闭与信号转发，不把子界面的具体 UI 或逻辑写进父脚本。

---

## 四、新功能开发流程

每次新增功能必须按以下步骤执行：

1. **先确认功能属于哪个模块**  
   对照第二节的目录归属，决定放在 `ui/`、`ui/main_menu/`、`ui/finance/`、`systems/` 或 `data/`。

2. **如果需要，新建文件夹**  
   若该模块尚无目录（如未来的 `systems/`），先建目录再建文件。

3. **创建独立脚本（及场景）**  
   新功能尽量用新场景 + 新脚本实现，通过信号或接口与现有代码通信。

4. **不要直接修改现有大文件**  
   若现有脚本已很大或职责已很多，应通过新模块扩展，而不是继续往该脚本里堆逻辑。

5. **写完后检查是否违反本规则**  
   用第五节的自检清单做一次核对，违反则先调整结构再继续。

---

## 五、自检机制（非常重要）

在**每次写代码之前**，必须先检查：

- [ ] **是否符合 PROJECT_RULES.md**  
  新代码的放置位置、职责划分是否与本文档一致。

- [ ] **是否应该拆成独立文件**  
  是否会导致单文件过长、多职责、UI 与逻辑混杂、一个界面管多个子界面；若是，先拆文件再实现。

- [ ] **是否把逻辑写进了不该写的地方**  
  例如：地图脚本只做地图与菜单切换；主菜单脚本只做主菜单；账单逻辑只在 `ui/finance/`；游戏系统只在 `systems/`。

若发现违反上述任一条，**需要先修正结构再继续开发**，不得为图省事而破坏规则。

---

## 六、总结

- **模块化**：功能按目录与文件拆分，职责清晰。  
- **地图纯粹**：world_map 只负责地图、城市、菜单/面板的打开与切换。  
- **先分后写**：新功能先定模块、再建文件、最后实现，并做自检。

开发过程中以本文档为准；若需调整规则，应同步更新本文档并保持与实现一致。
